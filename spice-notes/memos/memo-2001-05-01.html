Memo: Email between Steve Leach (SFKL) and Chris Dollin (CJD), Tuesday 01/05/2001.

  1. SFKL agreed to deprecate the use of "&" in MillScript in favour of a maplet operator, syntax to be decided. Suggestions are ":>" and ":-". Concern was expressed about avoiding symbols which small typing errors could turn into hard-to-diagnose problems.

    Aside: The symbol that was finally adopted was ==> which is understood to be ASCII-art for Unicode 21A6 (↦).

  2. CJD agreed to deprecate the use of "&" in Spice in favour of "++" as append operator.

  3. Both SFKL & CJD agreed to use "++" as append operator i.e. 'polymorphic concatenation'. We are both agreed on the need for the arguments to be type-compatible.

  4. The syntax for holes has been fixed as "?" or "?n" where n is a small positive integer. If any hole is labelled by a number, all holes in the same expression must also be labelled. (i.e. ?2 + ? is not legal.) The "^" operator is reserved. The intended use is denoting match variables.

    Furthermore, it is agreed that "?" is able to deal with short-circuit operators as well. In particular, "? and ?" means the functional version of boolean and. Compilers are expected to generate code that avoids needless copying when all the arguments are holes.

  5. CJD agreed to replace the postfix type operator "?" with "??". "type??" is intended to mean the anonymous union of type with Absent.

  6. The "$" symbol is to be reserved. The intended use is accessing environment variables.

  7. SFKL agreed to augment the newMap operator to permit initialization using maplets & other maps. Furthermore there will be another constructor with the form 

    	alistToMap( k1, v1, k2, v2, ..., kN, vN ) -> map

  8. Both agreed that "%" and "//" should be reserved. Roles not determined. CJD pointed out that "/:" is currently allocated the roles of "fractional" division and that perhaps "//" should take over that role.
